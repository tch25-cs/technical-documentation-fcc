Overview

This document provides a structured approach to securing modern web applications and their supporting infrastructure. It is intended for developers, system administrators, and security analysts who are responsible for protecting critical assets.

  • Purpose: Establish a baseline of cybersecurity practices.
  • Scope: Authentication, encryption, logging, and incident handling.
  • Audience: Technical teams with operational responsibilities.
  • Assumptions: Linux servers, web stack (nginx, Node.js, or similar).
Tip: Security documentation should be practical. Use clear steps, tested commands, and reference frameworks.

Threat Model

Understanding what we are defending against is the foundation of security. Assets, threats, attack vectors, and mitigations are listed below:

Asset: Authentication Tokens
Threat: Token theft via XSS or logging
Impact: High
Mitigation: HttpOnly + Secure cookies, CSP, centralized vault

Asset: User PII (Personally Identifiable Information)
Threat: Exfiltration via misconfigured S3 bucket
Impact: Critical
Mitigation: IAM SCPs, server-side encryption (SSE-KMS), least privilege

Asset: Service Availability
Threat: Distributed Denial of Service (DDoS)
Impact: High
Mitigation: WAF rules, CDN-based DDoS protection, autoscaling

Controls

Controls should be mapped to established frameworks like NIST CSF, NIST SP 800-53, and OWASP. Example baseline controls include:

  • IA-2 – Enforce multi-factor authentication for privileged accounts.
  • SC-13 – Use TLS 1.2+ for all external communications.
  • AU-6 – Centralize log collection and enable automated alerting.
  • CM-3 – Require peer review for all infrastructure changes.
Warning: Unmapped controls may introduce gaps in compliance reporting.

Implementation

Security controls must be enforced with concrete configurations. Examples:

# Harden firewall with UFW
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow ssh
sudo ufw enable
# Enforce TLS 1.2+ in nginx
ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers HIGH:!aNULL:!MD5;
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

Security Headers

Harden HTTP responses with headers that mitigate common web risks:

HeaderPurposeExample
Content-Security-Policy (CSP) Mitigate XSS by restricting sources default-src 'self'; img-src 'self' https:; script-src 'self'
X-Content-Type-Options Prevent MIME sniffing nosniff
Referrer-Policy Limit referrer leakage strict-origin-when-cross-origin
Permissions-Policy Disable unused browser APIs camera=(), microphone=(), geolocation=()
Strict-Transport-Security (HSTS) Enforce HTTPS in browsers max-age=31536000; includeSubDomains

Testing & Validation

Controls should be verified regularly through manual checks and automation.

  • Run nmap scans to confirm exposed ports.
  • Validate security headers with curl.
  • Audit dependencies with npm audit or pip-audit.
  • Automate with CI/CD to block deployments if checks fail.
# Example: Check for key security headers
curl -sI https://example.com | grep -E "Strict-Transport-Security|Content-Security-Policy"

Incident Response

A structured process ensures consistent handling of security incidents:

  1. Detect: Alert triggered in SIEM.
  2. Triage: Classify severity (Low, Medium, High, Critical).
  3. Contain: Disable compromised accounts, block malicious IPs.
  4. Eradicate: Patch vulnerability, rotate credentials.
  5. Recover: Restore systems and validate integrity.
  6. Lessons Learned: Conduct post-mortem and update playbooks.

Severity Levels
